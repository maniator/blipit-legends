> This section covers Data model (all collections), roster-edit lock, routing per version, schedule generation, per-game seed derivation, and concurrency model. Part 2 of 6.

## Core Invariants
- All teams (built-in, user-created, autogenerated) live in existing customTeams collection; no leagueTeams collection
- Only one seasons doc with status='active' at a time (decision #8)
- seasons.featureFlags snapshot frozen at creation; flipping mid-season is no-op
- seasons.awards[] MUST be sorted by awardKey lex-ascending before every write (byte-stability for bundle sig)
- expectedWins is per-team on seasonTeams, NOT a per-season scalar
- seasonGames.derivedSeed cached at schedule generation; schema migrations MUST NOT recompute or rewrite it
- All PRNG-affecting iteration MUST use sorted arrays; never Object.keys, Set/Map iteration, or RxDB query order

## Roster-Edit Lock Contract
- customTeams doc locked when: seasons.status='active' AND customTeamId in that season's leagues[].teamIds
- Lock enforced at two layers: (1) CustomTeamEditor renders disabled with StatusBanner + deep-link; (2) customTeamStore.update() throws CustomTeamLockedError unless withSanctionedCustomTeamWrite context present
- CustomTeamEditor mode="locked": inputs disabled, DnD disabled, destructive actions hidden, editorReducer actions no-ops
- Sanctioned paths only: IL moves → seasonPlayerState+rosterSnapshot; trades → same; lineup/rotation → rosterSnapshot

## Schedule Generation
- Pure module; inputs frozen at creation; sub-PRNG: mulberry32(parseInt(fnv1a(`${masterSeed}:schedule`),16)>>>0)
- Odd team counts: byes; distribution deterministic; Mini preset (8 teams, 30 games, 3-game series) caveat: ~4.28 games/opponent — non-uniform opponent frequency, surface in wizard
- Sub-PRNG registry: :schedule (v1); :autogen:${autogenSubSeed} (v1); per-game derivedSeed cached; :trades:day:${gameDay} (v3); :tiebreak:${sortedTiedIds} (v3)

## Per-Game Seed Derivation (non-negotiable)
- deriveScheduledGameSeed(seasonId, seasonGameId): fnv1a(`${seasonId}:${seasonGameId}`) → parseInt(hex,16)>>>0 → .toString(36)
- reinitSeed: parseInt(cachedDerivedSeed, 36)>>>0 → mulberry32 state
- NEVER pass raw colon-joined string to reinitSeed (colon not base-36-safe)
- Cached on seasonGames.derivedSeed at schedule generation; sim reads cache
- ESLint rule (v1): restricts reinitSeed imports to sim entry point and league sim loop only

## Simulation Modes
- Watch/Manage: /game?seasonGameId=X; manager-mode available; result reconciled to seasonGames on completion
- Quick Sim (headless): single-game no-UI; headless game <50ms
- Sim to next user game: batch sequential; typically <16 games (one game day, Mini league)
- Sim full season (v4): fairness budget ~4 games per requestIdleCallback; progress bar; user can cancel
- Sequential by design: global mulberry32 instance; parallel sim = non-deterministic PRNG interleave

## Concurrency Model (binding)
1. Per-game claim: atomic status: scheduled→in_progress with claimedBy writer-id (crypto.randomUUID() per tab)
2. Batch mutex: per-seasonId in-memory mutex token; released on completion or abort
3. Watch ↔ headless mutual exclusion: active /game?seasonGameId=X blocks headless batch for same season
4. Multi-tab: multiInstance: false in RxDB config; stale in_progress rows reset to scheduled on app start
5. reinitSeed is LAST step before sim entry — after all async RxDB reads resolve

## Per-Game Flow (atomic)
1. Claim: status: scheduled→in_progress; abort if already in_progress/completed
2. Read derivedSeed; load rosterSnapshot; build seasonModifiers; reinitSeed(derivedSeed) — last step
3. Inject modifiers into gameplay context — injection contract FROZEN in v1; v2/v3/v4 only add fields
4. Run game to completion; (v2+) run injury rolls after box-score in fixed iteration order
5. Single RxDB bulk-write transaction; seasonTeams aggregates DERIVED (recompute from seasonGames, not incremented)
6. Advance seasons.currentGameDay when all games for that day complete

- location.state explicitly rejected (unreliable across reloads)
