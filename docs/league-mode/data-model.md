# League Mode — Data Model

> Companion to [`README.md`](README.md). All schemas obey the contracts in [`docs/rxdb-persistence.md`](../rxdb-persistence.md): indexed fields appear in `required[]`. **Migrations note:** every league/season collection is **brand new** in this subsystem — no existing user data needs structural migration. When a later phase (v2/v3/v4) adds optional fields to a brand-new collection, the only migration required is an identity strategy with documented defaults so RxDB's version check is satisfied; no data transformation runs. The single exception is the pre-existing `customTeams` collection (see §customTeams below + [`agent-prompts/v1.md`](agent-prompts/v1.md)).

## Single-library invariant

**All teams** — built-in, user-created, autogenerated for a league — live in the **existing `customTeams` collection.** League state references teams by `customTeamId`. There is no `leagueTeams` collection and no alternate team type. This is the most important invariant in the whole subsystem; revisit [`decisions.md`](decisions.md) #7 and #A3 before proposing anything that splits the team library.

Implications:

- The setup wizard's "autogenerate" action stages generated teams in wizard reducer state; the **single transactional write** to `customTeams` happens at Step 6 (Review & create) — see [`setup-wizard.md`](setup-wizard.md). Once persisted, generated teams appear in the user's library, can be edited via the existing Custom Team Editor (subject to the roster-edit lock once the season is `active`), can be exported/imported, and can be reused in future leagues.
- The Custom Team Editor schema (positions, batting/pitching blocks, sanitization) is the **only** team schema. League features extend `customTeams` only via additive optional fields (e.g., `autogen` marker), never with new `type` discriminators.
- Removing a team from a league does **not** delete the `customTeams` doc.

## Roster-edit lock contract

A `customTeams` doc is **locked from direct edits** while it satisfies all of:

1. There exists at least one `seasons` doc with `status === 'active'`, AND
2. that season's `leagues[].teamIds` includes this `customTeamId`.

Locked docs are:

- **Read-only in the Custom Team Editor route.** Form is rendered but disabled with a banner; deep-link to the season's team-management page is shown.
- **Write-rejected at the storage layer.** A `customTeamStore.update()` (or any equivalent write API) called against a locked doc throws `CustomTeamLockedError` unless the call is tagged with a sanctioned-write context (see below).

Sanctioned mutation paths — these set the sanctioned-write context and bypass the storage lock:

| Path                                      | Version | What it writes                                                                                                                                                        |
| ----------------------------------------- | ------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| In-season trade engine                    | v3      | Updates `seasonPlayerState.seasonTeamId` and `seasonTeams.rosterSnapshot`. **Does not touch `customTeams`.**                                                          |
| IL move (injury / recovery)               | v2      | Updates `seasonPlayerState.injuryStatus` and `seasonTeams.rosterSnapshot`. **Does not touch `customTeams`.**                                                          |
| Lineup / rotation update inside season UI | v2      | Updates `seasonTeams.rosterSnapshot`. **Does not touch `customTeams`.**                                                                                               |
| Minor-league call-up / demotion           | v4      | Updates `seasonTeams.rosterSnapshot` and `customTeams.minorLeaguePlayerIds[]`. The `customTeams` write is sanctioned.                                                 |
| Offseason carryover write-back            | v4      | Writes back current `rosterSnapshot` into `customTeams`. Runs only **after** the season has transitioned to `complete`/`abandoned` (so the lock is already released). |
| Setup-wizard autogen team creation        | v1      | Creates a new `customTeams` doc. The new doc is not yet in any active league, so no lock applies at creation time.                                                    |

Notably, **none of the v1–v3 sanctioned paths write to `customTeams` itself** — they all live in season-scoped state (`seasonTeams.rosterSnapshot`, `seasonPlayerState`). The customTeams doc remains the team's "permanent self" while season state captures in-flight changes. v4 carryover is the only path that ever propagates season changes back to `customTeams`, and it runs after lock release.

Implementation:

- A `withSanctionedCustomTeamWrite(context, fn)` helper wraps every sanctioned write site, threading the context through to `customTeamStore`. The store checks `context != null` before bypassing the lock.
- Lock check is a fast indexed query: `seasons.find({ selector: { status: 'active' } }).exec()` → flatten `leagues[].teamIds` → `Set<customTeamId>`. Cached per write batch.
- Write rejection logs via the shared logger and throws a typed error so UI can show a friendly message.

This contract is reaffirmed in the user-facing [`README.md`](README.md) non-negotiable contracts section.

### `seasonTransactions` (**v2** — introduced for IL events; expanded in v3 for trades)

One document per transaction (IL move, trade, call-up, demote). Introduced in **v2** to back the IL move events in the wear/injury system and the v2 transactions feed UI; the schema is forward-compatible with the v3 trade kinds (no v2→v3 schema bump on this collection).

| Field       | Type                                                                                                                                                                                                                                                              | Idempotency notes                                                                                                                                                                                                                                                                               |
| ----------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `id`        | string (primary)                                                                                                                                                                                                                                                  | For trades, derived deterministically as `trade:${proposalId}` where `proposalId = fnv1a(seasonId + gameDay + sortedTeamPair + sortedPlayerIds)`; for IL events as `il:${seasonPlayerStateId}:${gameDay}:${kind}`. Re-running a trade proposal collides on insert and is detected as duplicate. |
| `seasonId`  | string (indexed)                                                                                                                                                                                                                                                  |                                                                                                                                                                                                                                                                                                 |
| `gameDay`   | number (indexed)                                                                                                                                                                                                                                                  |                                                                                                                                                                                                                                                                                                 |

| `payload`   | object — kind-specific structure (e.g., trade asset lists, IL injury kind)                                                                                                                                                                                        |                                                                                                                                                                                                                                                                                                 |

### `seasonAwards` (v4)

One document per (season, award).

| Field                | Type                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| -------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `id`                 | string (primary) — `${seasonId}:${awardKey}`                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| `seasonId`           | string (indexed)                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| `awardKey`           | string — e.g. `'mvp_league_a'`, `'cy_young_league_b'`, `'reliever_of_the_year_league_a'`, `'rookie_of_the_year_league_b'`, `'manager_of_the_year_league_a'`                                                                                                                                                                                                                                                                                                                          |
| `winnerPlayerId`     | string \| null — set for player awards (MVP, Cy Young, Reliever of the Year, Rookie of the Year). Null for MoY.                                                                                                                                                                                                                                                                                                                                                                      |
| `winnerSeasonTeamId` | string \| null — set for team awards (Manager of the Year only). Null for player awards. Exactly one of `winnerPlayerId` / `winnerSeasonTeamId` is non-null per doc, determined by `awardKey`.                                                                                                                                                                                                                                                                                       |
| `formula`            | object — composite breakdown so the awards screen can show the math. **Shape is not RxDB-schema-enforceable** (RxDB JSON Schema does not constrain arbitrary nested objects). Documented contract: `{ components: { [termName: string]: number }, total: number, rank: number }`. Enforced at write-time by a `validateSeasonAwardFormula(doc)` helper called at every `seasonAwards` insert site, plus a unit test that asserts the renderer fails loudly on a malformed `formula`. |

> **`seasonAwards` XOR invariant.** "Exactly one of `winnerPlayerId` / `winnerSeasonTeamId` is non-null per doc, determined by `awardKey`" cannot be expressed in RxDB JSON Schema (`oneOf` is discouraged in RxDB schemas and does not compose cleanly with `required[]`). Enforce via:
>
> 1. A `validateSeasonAward(doc)` helper called at every insert site (rejects writes that violate the XOR or where the populated field doesn't match the `awardKey`'s expected kind — player vs team award).
> 2. A unit test that loops every `awardKey` and asserts the correct field is populated and the other is `null`.
>
> Owner: `@rxdb-save-integrity`.

### `seasonArchives` (v4, only if measured doc count exceeds budget)

Compressed archive of one completed season's `seasonGames` + `seasonTransactions` + `seasonPlayerState`.

| Field        | Type                           | Notes                                                                                                                                                                                                                                                                                  |
| ------------ | ------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `id`         | string (primary, = `seasonId`) |                                                                                                                                                                                                                                                                                        |
| `compressed` | base64 string (gzipped JSON)   |                                                                                                                                                                                                                                                                                        |
| `checksum`   | string                         | FNV-1a hex of `canonicalJSON(archivedData)` (see `src/features/saves/storage/canonicalJSON.ts`) **before** gzip — plain `JSON.stringify` has no guaranteed key-order stability across runtimes. Verified on every read; mismatch surfaces a corruption error without deleting the doc. |
| `archivedAt` | number                         |                                                                                                                                                                                                                                                                                        |

`seasons` and `seasonAwards` are **not** archived — they remain queryable for the seasons list & history UI.

## Additive fields on existing collections

### `customTeams` (additive — schema bump only when these land)

| Field             | Type                                                                   | Version added | Notes                                                                                                                                                                                                                                                                                                                                                          |
| ----------------- | ---------------------------------------------------------------------- | ------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `activeLeagueIds` | `string[]`                                                             | v1            | Teams can sit in multiple historical seasons. This is a **denormalized cache** for fast filtering only — correctness-critical reads (e.g., edit-lock checks) MUST re-derive from `seasons.leagues[].teamIds`, never trust this cache. Written in the same transaction that activates a season; cleared when a season transitions to `complete` or `abandoned`. |
| `autogen`         | `{ version: number; theme: string; parity: string; baseSeed: string }` | v1            | Present **only** on autogenerated teams. These fields are **optional** so existing customTeams docs continue to validate. Migration strategy v(N) → v(N+1) leaves the keys **omitted** on docs that don't have them (RxDB / persisted JSON cannot reliably round-trip an `undefined` value, so we never write the key with an `undefined` value — code that consumes these fields treats "key missing" as the absence signal).

## Determinism notes

- `seasonGames.derivedSeed` is computed on schedule generation, not at game-launch time. This guarantees the seed is stable even if `deriveScheduledGameSeed`'s implementation changes later.
- `seasonTeams.rosterSnapshot` freezes the roster at season start. Mid-season team edits in the customTeams editor do not leak into the active season.
- `seasons.featureFlags` snapshot guarantees a season replay sees the same flag values as when it was created.
